# 写真から一貫したデフォルメキャラクターを作るスキル 実装計画 v0.1

作成日: 2026-07-14  
関連資料: [デフォルメキャラクター一貫生成システム 設計書・運用アイデア・制作フロー](./2026-07-14-character-consistency-workflow.md)  
想定基盤: ChatGPT Images 2.0 / GPT Image 2  
計画の対象: 写真から人物・動物を独自のデフォルメキャラクターに変換し、角度・表情・ポーズを変えても同一キャラクターと同一作風を保つ再利用可能なスキル

---

## 1. 推奨結論

### 1.1 360度資料は有効だが、それだけでは足りない

連続した360度動画や大量の角度画像を毎回渡す必要はない。まずは次の5方向を、一枚の承認済みターンアラウンドにまとめる。

1. 正面
2. 3/4前
3. 真横
4. 3/4後ろ
5. 真後ろ

後ろ姿や横姿を基準化するため、服の背面、後頭部、耳、しっぽ、左右非対称の模様をモデルが勝手に補完する余地を減らせる。ただし、OpenAI公式ドキュメントも、複数回の生成で繰り返し登場するキャラクターやブランド要素の視覚的一貫性を維持できない場合があると明記している。したがって、ターンアラウンドは「保証」ではなく「成功率を上げる参照情報」と扱う。

### 1.2 最も効率がよい中核は Character Identity Package

一番重要なのは、画像を大量に作ることではなく、次の5要素を一つのバージョン付きパッケージにすること。

- 構造化されたキャラ仕様
- 少数の承認済み基準画像
- 固定・可変・禁止要素
- 用途別の参照画像選択ルール
- 生成後の一貫性QA

このパッケージを以下では `Character Identity Package`（CIP）と呼ぶ。

### 1.3 スキルは「モデル」ではなく「制作・検査ワークフロー」にする

スキルだけで画像モデルを再学習するわけではない。スキルの役割は次のオーケストレーションである。

1. 元写真から人物・動物の特徴を抽出する
2. ユーザーとデフォルメ方向を確定する
3. CIPを作成・版管理する
4. 指示に合わせて必要最小限の参照画像を選ぶ
5. 「変えるもの」と「維持するもの」を分けたプロンプトを組み立てる
6. 画像を生成または編集する
7. 出力を仕様書と照合し、不合格なら修正または再生成する

### 1.4 厳密な一貫性が必要なら3Dまたは2Dリグを併用する

AI画像生成だけで、あらゆる角度・ポーズにおける完全な幾何学的一致を保証することはできない。最終的にゲーム素材、アニメーション、量産グッズなどで厳密な形状一致が必要になった場合は、承認済みキャラを以下のどちらかに展開する。

- 2Dパーツ分け＋ボーンリグ
- 簡易3Dモデル＋固定シェーダー

AIは表情案・ポーズ案・背景・仕上げに使い、形状の正解はリグや3Dモデルに持たせる。MVPでは不要だが、将来の一貫性上限を上げるための拡張先として残す。

## 2. 公式情報から確定できること

### 2.1 確認済みの事実

- ChatGPT Images 2.0の公式紹介には、ターンアラウンドや表情を含むキャラクター参照シート、複数シーンの連続性、同一性を保ったバリエーションの作例がある。
- GPT Image 2は、テキストと画像を入力し、画像生成と画像編集を行える現行の画像モデルとして案内されている。
- Image APIでは、一枚以上の参照画像を使った新規画像生成ができる。
- GPT Image 2は参照画像を常に高忠実度で処理する。API側で `input_fidelity` を変更することはできない。
- Responses APIは、会話の流れを保った複数ターンの画像生成・編集に対応する。
- OpenAI Academyは、複数のアップロード画像を使う場合、少数の画像に絞り、画像ごとの役割を順番で説明することを勧めている。
- 同ガイドは、編集時に何を変え、何を維持するかを明示し、小さな変更を段階的に行うことを勧めている。
- GPT Image 2のAPIには日付付きスナップショット `gpt-image-2-2026-04-21` がある。
- 公式の制約として、繰り返し登場するキャラクターやブランド要素の一貫性、厳密な構図配置に失敗する場合がある。
- GPT Image 2は現時点で透明背景出力をサポートしていないため、透過素材が必要なら別の切り抜き工程が必要になる。

### 2.2 事実ではなく、検証すべき設計仮説

以下は公式保証ではなく、このプロジェクトで比較テストする仮説である。

- 5方向ターンアラウンドは正面画像だけより、後ろ姿・横姿の合格率を高める。
- 一枚ずつ独立生成した5方向画像より、一枚の同時生成ターンアラウンドを承認してから切り出す方が相互矛盾を減らせる。
- キャラ仕様書全文を毎回投入するより、依頼に関係する固定条件だけを短くコンパイルする方が指示競合を減らせる。
- 編集履歴を何段も継ぎ足すより、承認済みの基準画像へ戻して毎回差分を作る方がドリフトを減らせる。
- 人間の見た目に関するQAと、服・耳・しっぽなど明確な固定要素のQAを分けると、失敗原因を特定しやすい。

## 3. 独自の作風定義

プロンプトや仕様書では「ねんどろいど風」のような他社ブランド名を中心にせず、欲しい造形を独自のパラメータへ分解する。OpenAI Academyも、特定ブランドの模倣より、一般化された独自に所有可能なデザインを勧めている。

例:

- 2.3〜2.7頭身
- 頭部は全高の約40%
- 輪郭線は均一で明瞭
- フラット塗り＋1段階の影
- 鼻と口は小さく記号化
- 手足は短いが、指示されたポーズが読める長さ
- 目は大きいが、人物・動物固有の形を残す
- 彩度を限定した6〜10色程度の基本パレット
- 素材感はフィギュアではなく二次元イラスト

数値は初期値であり、1キャラ目の比較生成で調整する。

## 4. Character Identity Packageの構成

### 4.1 ディレクトリ案

```text
character-packs/
└── <character-id>/
    ├── character.yaml
    ├── manifest.json
    ├── source/
    │   └── 利用許諾を確認した元写真
    ├── canon/
    │   ├── master-front.png
    │   ├── master-3q-front.png
    │   ├── turnaround.png
    │   ├── face-expression-sheet.png
    │   ├── costume-lock-sheet.png
    │   └── style-lock-sheet.png
    ├── details/
    │   ├── face.png
    │   ├── back-costume.png
    │   └── unique-markings.png
    ├── tests/
    │   ├── cases.yaml
    │   └── approved-baselines/
    └── outputs/
        └── <version>/<request-id>/
```

元写真と個人固有のCIPは、再利用スキル本体へ埋め込まない。スキルには空のテンプレートと処理手順だけを置き、個人データはプロジェクト側で分離管理する。

### 4.2 `character.yaml` の最小スキーマ案

```yaml
id: cat-red-hoodie
version: 1.0.0
subject:
  type: cat
  identity_features:
    - round face
    - triangular upright ears
    - dark marking above left eye
geometry:
  head_ratio: 0.42
  body_style: short-limbed chibi
  asymmetry:
    - left-eye marking exists only on the left
colors:
  fur: "#8A8178"
  outline: "#24201D"
costume:
  required_layers:
    - red hoodie
    - black shorts
    - gray sneakers
  back_details:
    - hood always visible
    - tail exits below hoodie hem
style:
  line: clean, uniform, dark outline
  fill: flat color
  shadow_levels: 1
locks:
  must_keep:
    - both ears
    - tail length and color
    - clothing layers and design
    - body proportion
  may_change:
    - expression
    - pose
    - gaze
    - background
  forbidden:
    - remove clothing
    - add hats
    - change species
```

YAMLだけを正解にせず、承認済み画像を視覚上の正解とする。文章で表現しにくい顔立ち・線・微妙な比率は画像を優先し、耳の数・服の構成・色など機械的に判定しやすい項目はYAMLを優先する。

## 5. 参照画像ルーター

毎回CIP全体を渡さず、依頼内容に合わせて2〜4点程度の参照へ絞る。枚数は固定値ではなく、OpenAI公式の「少数の明確な参照が扱いやすい」という原則に基づく初期方針である。

| 依頼 | 優先参照 | 補助参照 |
|---|---|---|
| 笑顔・泣き顔 | 顔アップ、正面基準 | 表情シート、スタイルシート |
| 横向き | 真横、3/4前 | 衣装シート、顔アップ |
| 後ろ向き | 真後ろ、3/4後ろ | 衣装背面、スタイルシート |
| 走る・跳ぶ | 最も近い角度の全身 | ターンアラウンド、衣装シート |
| 服は固定で小物追加 | 正面または対象角度 | 衣装シート、小物資料 |
| 漫画の複数コマ | 3/4前、顔アップ | ターンアラウンド、スタイルシート |

各画像の役割をプロンプト内で明示する。

```text
Image 1: identity and face reference.
Image 2: target viewing-angle reference.
Image 3: exact costume and rear-detail reference.
Image 4: line, color, and shading style reference.
```

## 6. 生成方式の選択ルール

### 編集を使う

- 表情だけを変える
- 背景だけを変える
- 小物を一つ加える
- 色や小さな描画ミスを直す

### 参照付き新規生成を使う

- 正面から後ろ向きへ大きく角度を変える
- 立ち姿から走り・ジャンプへ大きくポーズを変える
- 構図やカメラ距離を大きく変える
- 複数キャラクターの新しい場面を作る

「生成より編集の方が常に安定する」とは決めつけない。小さな変更は編集、大きな視点・骨格変更は対象角度の基準画像を使った新規生成としてA/Bテストする。

マスク編集は厳密な形状保証ではない。公式ガイドでもマスクはプロンプトベースのガイドとして扱われ、輪郭へ完全一致しない場合があるとされている。

## 7. プロンプトコンパイラ

長いキャラバイブルをそのまま毎回貼るのではなく、依頼に関連する条件だけから短いプロンプトを組み立てる。

### 構成

1. 出力目的
2. 変更する要素
3. 対象角度とポーズ
4. 維持する固定要素
5. 禁止要素
6. 作風と出力条件

### 例

```text
Create a rear three-quarter view of the same character, looking back with a broad smile.
Change only the pose, gaze, and expression. Preserve the same identity, 2.5-head body ratio,
both triangular ears, tail shape, red hoodie including its rear hood and white cords,
black shorts, gray sneakers, palette, clean uniform outline, flat fill, and one-level shadow.
Do not remove or redesign clothing, add accessories, change species, or alter the art style.
```

重要条件は短く、具体的に、依頼ごとに再掲する。これは公式ガイドの「何を変え、何を維持するかを明示する」「小さく段階的に修正する」という方針に合わせる。

## 8. 生成後QA

プロンプトで防ぐだけでなく、出力を検査して不合格を戻す。ここが服・耳・作風の崩れを減らす中核となる。

### 8.1 ハード失格条件

- 耳・しっぽ・手足など、必須部位の数または存在が違う
- 種別が違う
- 固定衣装が脱げている、欠けている、別デザインになっている
- 左右非対称の模様が逆または消失している
- 頭身が指定範囲から明らかに外れている
- 指定と違う角度・ポーズである
- 作風が写実・3Dフィギュア・別の線画へ大きく変化している

### 8.2 採点項目

各項目を0〜2点で採点する。

| 項目 | 0 | 1 | 2 |
|---|---|---|---|
| 顔・同一性 | 別キャラ | 一部違和感 | 基準と一致 |
| シルエット・頭身 | 大きく違う | 小さな差 | 一致 |
| 耳・しっぽ・模様 | 欠落・誤り | 軽微な差 | 一致 |
| 衣装 | 欠落・変更 | 細部差 | 一致 |
| 作風 | 別作風 | 軽微な差 | 一致 |
| 指示達成 | 未達 | 部分達成 | 達成 |

ハード失格がなく、合計10/12点以上を初期の合格候補とする。この閾値は品質目標であり、モデル性能に関する事実ではない。1キャラ目の20〜30ケースで妥当性を調整する。

### 8.3 修正ループ

1. 失敗項目を一つに分解する
2. 小さな編集で直せるか判定する
3. 直せる場合は「変更はその項目だけ」として編集する
4. 同一性や大きな骨格が崩れた場合は、承認済み基準へ戻して再生成する
5. 同じ失敗を2回繰り返したら、参照画像の選択またはCIPの曖昧さを見直す

生成物を次の生成物の唯一の基準にし続けない。承認済みCanonへ定期的に戻す。

## 9. スキル本体の構成案

スキル名候補: `create-consistent-character`

```text
create-consistent-character/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── references/
│   ├── character-schema.md
│   ├── prompt-rules.md
│   ├── reference-routing.md
│   └── qa-rubric.md
├── scripts/
│   ├── validate_character_pack.py
│   ├── select_references.py
│   ├── build_prompt.py
│   └── record_evaluation.py
└── assets/
    └── templates/
        ├── character.yaml
        └── test-cases.yaml
```

### `SKILL.md` に残すもの

- 発動条件
- 全体ワークフロー
- ユーザー承認が必要な地点
- 参照ファイルを読む条件
- 生成・編集・QAの順序

### 参照ファイルへ分離するもの

- 人物用・猫用・マスコット用の詳細スキーマ
- プロンプト例
- 角度別の参照画像選択規則
- QA基準と失敗例

### スクリプトに任せるもの

- 必須項目とファイルの検証
- リクエストに応じた参照画像候補の列挙
- 構造化仕様からのプロンプト組み立て
- テスト結果のCSV/JSON記録

個人の元写真、承認済みキャラ画像、APIキーはスキルへ同梱しない。

## 10. MVP制作計画

期間は作業量の仮見積もりであり、APIの生成待ち時間と人間の承認回数により変わる。

### Phase 0: 成功条件の確定（半日）

- 最初の対象を一人または一匹に限定する
- 主用途を一つ決める（例: アプリの立ち絵）
- 固定衣装を一着に限定する
- 5つの試験ポーズと4つの試験表情を決める
- 写真本人または権利者の利用許諾を確認する

成果物:

- `MVP_SCOPE.md` 相当の合意事項
- テストケース一覧

### Phase 1: キャラ方向性の確定（1〜2日）

- 元写真の品質と不足角度を確認する
- 独自スタイル仕様を3案作る
- 同じ正面写真から3案を比較生成する
- 1案を選び、正面無表情のCanonを承認する

完了条件:

- 顔、頭身、輪郭、配色、衣装をユーザーが承認
- 「似ているが嫌な誇張」が明文化されている

### Phase 2: CIP作成（1〜2日）

- 5方向ターンアラウンドを作る
- 顔アップ、衣装ロック、スタイルロックを作る
- `character.yaml` を作る
- 各画像へ役割とバージョンを付ける

完了条件:

- 正面・側面・背面で服、耳、しっぽ、模様が矛盾しない
- 固定・可変・禁止項目が曖昧語なしで書かれている

### Phase 3: 手動ワークフロー検証（1〜2日）

- 角度5ケース
- 表情4ケース
- 動作5ケース
- 背景または小物3ケース

同じ依頼を以下で比較する。

1. 正面画像だけ
2. ターンアラウンド＋短い固定条件
3. 参照ルーター＋短い固定条件＋QA

完了条件:

- 提案方式が基準方式より明確に良いか、同等なら追加工程に見合わないと判断できる
- よくある失敗上位5件と修正方法が特定される

### Phase 4: スキルMVP実装（2〜4日）

- `skill-creator` の標準手順でスキルを初期化する
- CIP検証、参照選択、プロンプト生成、QA記録を実装する
- 画像生成は既存の `imagegen` 能力を呼び出す
- 一人または一匹のCIPを使って通し試験する
- `quick_validate.py` でスキル構造を検証する

完了条件:

- 「このキャラを後ろ向きで笑わせて」のような依頼から、参照選択、生成、QA、保存まで一続きで実行できる
- 不足資料がある場合に、勝手に補完せず何が必要かを返せる

### Phase 5: API化・自動QA（任意、4〜7日）

- 画像生成をGPT Image 2 APIへ接続する
- 再現性を重視する場合は日付付きモデルスナップショットを使う
- QA結果と生成条件を記録する
- 不合格時の限定編集またはCanonからの再生成を自動化する
- 透過が必要なら背景除去工程を追加する

完了条件:

- 同じ回帰テストを繰り返し実行できる
- モデル・CIP・プロンプトのバージョンから出力を追跡できる

## 11. ChatGPT運用とAPI運用の選択

### まずChatGPT Images 2.0を使う理由

- 人間が結果を見ながら方向性を固めやすい
- キャラの好みを言葉にする探索段階に向く
- 画像の比較・小さな修正を対話的に進められる

### 本番でImage APIを検討する理由

- GPT Imageモデルを直接指定できる
- 日付付きスナップショットでモデルバージョンを固定できる
- 入出力、参照画像、CIP版、QA結果を記録しやすい

### Responses APIを検討する場合

- 会話的な複数ターン編集が製品要件の場合
- 画像IDや会話コンテキストを使った編集体験が必要な場合

ただし、Responses APIの画像生成ツールは画像モデル選択を内部で扱うため、画像モデルの明示的な版固定を最優先する場合はImage APIを比較候補にする。

## 12. リスクと対策

| リスク | 対策 |
|---|---|
| キャラが毎回少しずつ変わる | Canonへリベース、参照ルーター、QA、版管理 |
| 服が脱げる・変わる | 衣装の前後資料、必須レイヤー、ハード失格 |
| 耳・しっぽが消える | 角度別資料、部位数のハード失格、再生成 |
| 参照画像同士が矛盾する | 一枚のターンアラウンドを承認後に切り出す |
| プロンプトが長く競合する | 依頼に関連する条件だけをコンパイルする |
| 長い編集履歴で作風が漂う | 派生画像ではなく承認済みCanonから再開する |
| モデル更新で出力傾向が変わる | APIスナップショット、回帰テスト、CIP版管理 |
| 本人の写真を無断利用する | 本人・権利者の許諾確認、元写真の分離管理 |
| 特定ブランド模倣に依存する | 形状・線・色・塗りを独自仕様として定義する |
| 透過PNGが直接得られない | 単色背景＋別工程での背景除去 |

## 13. 最初の検証セット

1キャラ目では以下だけを試す。大量展開は合格基準が固まってから行う。

### 角度
